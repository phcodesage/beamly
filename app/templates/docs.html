Overview

Beamly is a modern, production-quality peer-to-peer file transfer example. By utilizing browser-native WebRTC standards, Beamly achieves true browser-to-browser transfers:

100% Serverless Payloads

Your files are never uploaded, cached, or buffered on the backend server. They stream directly from disk to disk.

Mandatory Encryption

All data streams are encrypted out-of-band using DTLS-SRTP, rendering payloads unreadable to middleboxes or hosting routers.

How to Use Beamly

  1. Create a Room: Click the CREATE TRANSFER ROOM button on the landing page to initialize a single-use room.
  2. Connect Devices: Share the generated room link or scan the dynamic SVG QR code on a second browser (e.g. your phone).
  3. Queue and Transfer: Once the connection state shifts to CONNECTED, drag-drop files or paste screenshots directly from your clipboard.
  4. Save Files: The receiver will dynamically reconstruct incoming chunks in the background and reveal a yellow Save button to download the file locally.

WebRTC & Signaling

Before two browsers can stream data directly, they must coordinate how to reach each other. This is the only responsibility of the Flask server, which coordinates signaling events using Socket.IO:

// Signaling lifecycle handshake stages:

1. Client A joins room -> server assigns role "initiator"

2. Client B joins room -> server assigns role "joiner"

3. Server broadcasts "peer_joined" to Client A

4. Client A creates RTCPeerConnection, mounts DataChannel, creates SDP offer

5. Client A sends SDP offer to Server -> Server relays to Client B

6. Client B receives offer, applies it, creates SDP answer -> Server relays to Client A

7. Both exchange ICE Candidates locally gathered via STUN servers to complete routing path

💡 ICE Candidate Queueing: To prevent race conditions, incoming ICE candidates received before the remote session description is fully configured are buffered in an array and flushed immediately after description binding completes.

Chunking & Backpressure

WebRTC data channels work best when files are split into small, manageable chunks rather than sent as one large buffer. Beamly divides files into **64 KiB chunks** on-the-fly.

Solving Buffer Overflows (Backpressure)

Browsers queue outgoing messages in an internal buffer. If a file is loaded and sent too fast, the socket buffer overflows, causing connection dropouts. Beamly implements a high/low water backpressure listener:

// Backpressure monitoring flow inside sender.js

const HIGH_WATER_MARK = 1 * 1024 * 1024; // 1 MiB

const LOW_WATER_MARK = 64 * 1024; // 64 KiB

while (offset < file.size) {

  if (channel.bufferedAmount > HIGH_WATER_MARK) {

    // Buffer full. Await the low water event listener before proceeding!

    await new Promise(resolve => {

      channel.addEventListener('bufferedamountlow', resolve, { once: true });

    });

  }

  // Slice and send next chunk...

}

Security & Privacy

Privacy is a first-class citizen in Beamly's design:

  • No Logs: No databases store transfer transactions. Session logs are transient inside server console memories.
  • Ephemeral Rooms: Active room keys are deleted from memory the second both devices close the browser window.
  • Out-of-Band Keys: WebRTC keys are negotiated directly between endpoints, ensuring intermediate proxies cannot inspect payloads.
  • XSS / Path Traversal Protections: Filenames are sanitized on the receiver end using regex filters to strip directory traversal injections (`../`).

Developer Setup

If you are running the project locally or preparing to contribute, execute the following commands:

# 1. Clone and enter directory

git clone git@github.com:phcodesage/beamly.git

cd beamly

# 2. Build Tailwind CSS files

npm install

npm run build:css

# 3. Create virtual environment and install packages

python3.11 -m venv venv

source venv/bin/activate

pip install -r requirements.txt

# 4. Launch the application

python run.py

For contribution guidelines and PR structures, check out our CONTRIBUTING.md file.
